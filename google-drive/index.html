Using the dashboard

Panel reference

What each panel is for, how to read it, and the action it should drive. The three objectives read left to right as an escalation path: who is active and where exposure is growing → what those users did to DLP-flagged data → where that sensitive data lives and who owns it.

How the filters work

The fieldset sets tokens that cascade into every panel, so the whole board re-scopes together. Time range sets the window. Index selects where Google Workspace data lands. Org unit (from gws_users.csv) scopes to one department. Data type narrows Objective 3 to specific DLP detectors. Sensitivity sets a severity floor. One interaction sits outside the fieldset: clicking a row in Top users by Drive activity sets a user token that filters the Objective 2 panels into a per-user investigation view.

Objective 1 · Visualize user access
source: gws:reports:drive · gws:reports:login · lookup gws_users.csv
Access eventssingle value

Purpose: a top-line pulse of Drive activity — view, edit, download, preview, print, copy.

Tells you: overall interaction volume; a swing versus trend is the first cue something unusual is happening before you drill into who or what.

Active userssingle value

Purpose: distinct people who touched Drive in the window.

Tells you: how broad the activity base is. Paired with Access events, it separates "a few users doing a lot" from "everyone doing a little" — a spike in events with flat user count points at one actor or an automated process.

External access eventssingle value

Purpose: events where files were shared or made reachable outside the domain. Magenta because it is the likeliest risk on this row.

Tells you: the size of the external-exposure surface right now and whether it is growing — the number a reviewer watches most closely here.

Failed loginssingle value

Purpose: authentication context — failed, challenged, and suspicious logins.

Tells you: whether access is happening against a backdrop of credential pressure; a rise here alongside unusual Drive access is a different story than clean activity.

Drive access by event type over timeline chart

Purpose: trends view, download, and edit hour by hour.

Tells you: the rhythm of normal activity and the shape of any anomaly — an off-hours download spike or a burst against a flat baseline. The download line is the one to watch for exfiltration patterns.

Internal vs external visibility changespie

Purpose: splits sharing/visibility-change events into internal versus external.

Tells you: how much sharing crosses the domain boundary. A rising external slice is an early indicator of data leaving controlled hands, independent of whether files are sensitive yet.

Top users by Drive activitytable · click to filter

Purpose: ranks the busiest users by activity and distinct files touched, with org unit.

Tells you: who the heaviest users are and whether any touch an unusually wide file set. Clicking a row filters Objective 2 to that person — the primary pivot from "who is active" into "what did this user do to sensitive data."

Access by org unitbar chart

Purpose: aggregates access volume by department via the user→org-unit lookup.

Tells you: which parts of the org generate the most activity — context for the sensitive-data panels below. Heavy Finance or Legal volume is normal; the same from an unexpected team is a question.

Objective 2 · Monitor actions on sensitive units
source: gws:reports:drive · gws:reports:token · joined to sensitive_docs.csv (DLP-scored docs only)
Sensitive downloadssingle value

Purpose: downloads of DLP-flagged documents. Magenta — the clearest exfiltration signal on the board.

Tells you: how much sensitive content is leaving Drive as local copies, and by how many users. A climb here most often starts an investigation.

Sensitive external sharessingle value

Purpose: distinct sensitive documents whose visibility changed to external or anyone-with-link.

Tells you: how many sensitive files were newly exposed outside the domain. Where downloads are "copied out," this is "left the door open" — and it often persists silently.

Sensitive deletes / trashsingle value

Purpose: delete and trash actions on flagged documents.

Tells you: whether sensitive records are being removed — relevant to insider risk (covering tracks) and to retention obligations where deletion itself may be a violation.

3rd-party apps with Drive accesssingle value

Purpose: distinct OAuth apps currently holding Drive scopes.

Tells you: the size of the third-party / shadow-IT surface that can read Drive data — an access path that bypasses the human-actor panels entirely.

Risky actions on sensitive documentstable

Purpose: the detail behind the single values — who did what to which sensitive document, with data type, severity, last-seen, and count.

Tells you: the specific, triageable events. A named user, named document, PCI tag, critical severity, timestamp — enough to decide whether to escalate.

Sensitive sharing & visibility changestable

Purpose: every sharing/visibility change on a sensitive document, with before/after scope and direction.

Tells you: exactly how exposure changed — "private → anyone-with-link," "domain → external user" — and the Direction column lets you scan straight to the external-bound changes.

Third-party OAuth apps with Drive scopetable

Purpose: the inventory behind the third-party tile — each app, scope, authorizing users, event volume, first-seen.

Tells you: which apps to trust and which to question. A full-drive scope on an unverified app authorized two days ago is exactly the row this exists to surface.

Objective 3 · Identify where sensitive information is stored
source: gws:reports:rules (DLP rule log events) · gws:reports:drive (label events)
DLP incidentssingle value

Purpose: total DLP rule violations in the window. Magenta — the headline risk metric for this objective.

Tells you: the overall pressure of sensitive-data detection right now and whether it is trending up.

Files with sensitive datasingle value

Purpose: distinct documents that triggered at least one DLP rule.

Tells you: the real footprint — how many actual files hold sensitive content, regardless of how many times each was detected. The number that answers "how big is our exposure."

Owners holding sensitive filessingle value

Purpose: distinct owners of the flagged documents.

Tells you: how concentrated ownership is. Few owners means accountability is easy to assign; many means sensitive data is spread thin and harder to govern.

Top detectorsingle value

Purpose: the single most-triggered data type in the window.

Tells you: the dominant category in play (e.g., PII), so the reviewer knows which regulatory lens to lead with before reading the breakdowns.

Where sensitive data lives — by org unitbar chart

Purpose: distinct sensitive files grouped by owning department.

Tells you: which teams are the custodians of the most sensitive data — the map for prioritizing controls and access reviews. "Shared / unassigned drive" rows flag ownerless sensitive data, a finding in itself.

Incidents by detectorpie

Purpose: the proportional mix of detector types — PII, PCI, SIN/national ID, PHI.

Tells you: the composition of the problem at a glance. A heavy PCI or PHI slice points the conversation straight at the relevant compliance regime.

DLP incidents over time by severitystacked column

Purpose: daily incident volume stacked by severity.

Tells you: whether detections are rising or falling — and whether the critical band is growing. A flat total hiding a growing critical layer is a deterioration the single values would miss.

Top documents triggering DLP rulestable · drill to detail

Purpose: ranks the documents generating the most incidents, with owner, org unit, data types, severity.

Tells you: the named files to remediate first; clicking a row opens the incidents in Search. The most actionable panel here — turns "we have sensitive data somewhere" into "fix these documents, owned by these people."

Sensitivity label distributiontable

Purpose: summarizes Drive classification-label activity — which labels and values, across how many documents.

Tells you: how the manual classification scheme is being used, complementing automated DLP. Gaps between labeled and DLP-detected documents reveal where policy and reality diverge.

Two boundaries for anyone using the board

Activity ≠ inventory. Objectives 1–2 only show files that were touched; Objective 3 reflects DLP scans at rest — so "where it lives" is complete even when "who touched it" is quiet.

Snippets stay in Google. DLP events carry detector type, severity, and document identity, not the matched content. Deep snippet review stays in the Workspace security investigation tool.

What the customer must have ingested

        
All panels read from the Splunk Add-on for Google Workspace (Splunkbase app 5556) on a heavy forwarder, using a GCP service account with domain-wide delegation and the admin.reports.audit.readonly scope.

      


      

        
1 · Reports-API inputs to enable

        
    
          
  • gws:reports:drive — Objectives 1 & 2 (access events, actions, sharing, label events)
    • 
          
  • gws:reports:login — Objective 1 auth context
    • 
          
  • gws:reports:token — Objective 2 third-party OAuth app access
    • 
          
  • gws:reports:rules — Objective 3 DLP rule log events. This is the rules applicationName in the Reports API. Confirm your add-on version exposes a rules input; if not, add a custom modular input or collect via the Google Workspace Data Manager. DLP requires Enterprise Standard/Plus, Frontline, Education, or Cloud Identity Premium on a Workspace license.

3 · Build the two lookups

        
sensitive_docs.csv — populated by the scheduled search above (run hourly). It is the single definition of "sensitive" that Objective 2 joins against, so actions are only flagged for documents DLP has actually scored. Key on doc_id.

        
gws_users.csv — pulled from the Directory API (the add-on's user input, or a scheduled export). Needs at minimum email and org_unit; add manager / group to extend the access panels.

      


      

        
4 · Two limits to set with the customer up front

        
Activity ≠ inventory. The Drive audit stream only shows files that were touched. DLP-scanned files appear in Objective 3 even if untouched, but won't generate Objective 1/2 activity until accessed.

        
Sensitive-content snippets stay in Google. The rules events carry detector type, severity, and document identity — not the matched content. Deep snippet review stays in the Workspace security investigation tool under the View sensitive content privilege.
